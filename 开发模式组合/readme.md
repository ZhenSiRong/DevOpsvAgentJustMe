## 推荐模式：后端优先 + TDD + 前后端分离并行

**核心逻辑：** 先把后端接口和核心链路跑通，前端基于真实 API 对接开发。不是"前端先出 mock 页面再等后端"，也不是"前后端同时开工各写各的"。

### 具体节奏

**第 1 阶段（你主导，约 3-5 天）：后端骨架 + 核心链路 TDD**

先不碰前端，专注把 Agent 的 6 段核心流程用 TDD 方式跑通：

1. **TDD 红绿循环写安全校验器**——先写测试（"输入 `rm -rf /` 应该返回 BLOCKED"），再写实现让测试通过。这是赛题评分的核心功能点，必须最先过。
2. **TDD 写探针模块**——测试驱动写 `probe.disk_usage()` / `probe.process_list()` 等，确保 subprocess 调用和输出解析可靠。
3. **搭 FastAPI 路由骨架**——把 `POST /chat` 的请求→Agent→响应这条主路径串起来，哪怕 LLM 还没接入也先用 mock 返回。
4. **接 LLM**——配 .env，跑通一次完整的 "用户输入→探针→LLM推理→校验→mock执行→返回" 循环。

这个阶段结束时，`curl http://localhost:8000/api/v1/chat -d '{"message":"清理垃圾"}'` 能返回一个结构正确的 JSON。

**第 2 阶段（并行启动，约 5-7 天）：你继续后端深挖 + 队友开始前端**

你做：
- 完善审计日志写入（audit_logs 表 CRUD）
- 实现最小权限执行引擎（devops-runner 切换）
- 会话管理 API 全套 CRUD
- WebSocket 端点

队友做（基于你第 1 阶段产出的真实 API 文档/Swagger UI）：
- Vue3 项目初始化（Vite + Tailwind）
- 对话交互区组件（Markdown 渲染 + 输入框 + 消息列表）
- 调 `POST /chat` SSE 流式对接真实后端
- 会话管理侧边栏（调 sessions CRUD）

关键点是**你先把 Swagger 文档稳定下来**，队友照着文档开发，不需要你们实时对齐。

**第 3 阶段（联调，约 2-3 天）：前后端合拢**

主要解决：
- SSE/WS 数据格式不一致
- 审计日志面板对接
- 错误状态展示（SECURITY_BLOCKED 等错误码的前端友好提示）
- 跨场景端到端测试（"清理垃圾"、"查看进程"、"重启服务"三个完整 happy path）

## 为什么不选其他模式

| 模式 | 为什么不适合 |
|------|------------|
| 前端优先 | 后端 API 还没定义，前端只能写死数据或 mock，后面改接口要返工 |
| 前后端完全并行 | 你们只有 2 人，没有多余的人力做接口契约先行定义和 mock server 维护 |
| 前后端联调一起开发 | 效率最低——一个人写 API 另一个人等着对接，大量空闲等待 |
| 不用 TDD | 安全校验器和执行引擎是高危代码（涉及系统命令执行），没有测试覆盖不敢上线 |

---

## TDD 在本项目中的具体应用位置

不是所有代码都 TDD，**只在以下 3 个模块强制 TDD**：

| 模块 | 测试先行理由 |
|------|------------|
| 安全校验器 | 直接关系系统安全，漏一条规则可能导致误删文件 |
| 执行引擎 | 涉及 subprocess 和权限切换，必须验证不会越权 |
| 探针解析器 | 不同 Linux 发行版的命令输出格式可能不同，需要测试覆盖 |

其他模块（路由层、前端 UI、数据库迁移）正常写就行，不需要红绿循环。

## 一句话总结

**后端 TDD 打底 → API 契约稳定 → 前端基于真实 API 并行开发 → 联调收尾**。这个模式在 2 人团队里效率最高，风险最低。
