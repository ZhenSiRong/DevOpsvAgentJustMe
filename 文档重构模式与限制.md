## 一、新模式：三层一入口

用三层目录结构替代当前根目录的 10 个散乱目录。

```
DevOpsAgentPlaning/
│
├── README.md                    ← 【总入口】唯一门面
├── docs-planning/               ← 【规划层】比赛评审文档
│   ├── README.md                ← 规划层导航页
│   ├── 01-business-goals.md
│   ├── 02-functional-requirements.md
│   ├── 03-core-workflow.mmd
│   ├── 04-data-model.md
│   ├── 05-api-contract.md
│   ├── 06-tech-stack.md
│   ├── 07-architecture.mmd
│   ├── 08-development-guide.md
│   └── 09-design-summary.mmd
│
├── devops-agent/
│   ├── README.md                ← 【工程入口】快速开始
│   ├── docs/
│   │   ├── README.md            ← 工程文档导航
│   │   ├── tui-usage-guide.md
│   │   ├── backend-gap-analysis.md
│   │   ├── selinux-policy-design.md
│   │   └── architecture-decisions.md
│   └── ...
│
└── assets/                      ← 【静态资源】图表、截图
    ├── architecture-diagram.png
    ├── core-workflow.png
    └── ...
```

### 新模式的三个原则

| 原则 | 说明 |
|------|------|
| **线性顺序** | `01` ~ `09` 数字前缀对应评审阅读顺序，评委从 `01` 顺读到 `09` 就能理解全貌 |
| **一层一入口** | 每个目录一个 `README.md`，告诉读者"这个目录里有什么、该按什么顺序看" |
| **规划与工程分离** | `docs-planning/` 是写给评委看的，`devops-agent/docs/` 是写给开发者看的，互不污染 |

## 三、重构限制规则（什么能动、什么不能动）

### 🔴 禁止做的事

| # | 限制 | 理由 |
|---|------|------|
| 1 | **禁止删除原始 mmd 文件** | mmd 是 mindmap 源码，后续可能还需要编辑。迁移时复制到 `assets/` 保留原始文件，旧目录里暂时保留直到确认新结构稳定 |
| 2 | **禁止修改 devops-agent/src/ 下的代码** | 本次重构只动文档，不动代码 |
| 3 | **禁止删除 .workbuddy/memory/** | AI 工作记忆，与项目文档无关但不可删 |
| 4 | **禁止合并 eddx 文件** | eddx 是亿图图示源文件，只能导出为图片使用，不能转成 markdown |
| 5 | **禁止删除旧目录直到新目录验证完成** | 先建新、验证链接有效、再删旧，防止误删 |

### 🟢 必须做的事

| # | 动作 | 理由 |
|---|------|------|
| 1 | **删除 AI 对话残留** | 接口契约和数据模型文档末尾的"要不要我把..."等句子必须删除 |
| 2 | **合并重复内容** | 原本+译本合并为一份；技术栈三层合并为一份；开发模式组合合并为一份 |
| 3 | **删除空目录** | `AAH架构设计/` 下的三个空目录直接删除 |
| 4 | **删除近乎空的文件** | `scorePie.mmd` (141B) 合并到功能清单主文档后删除 |
| 5 | **根目录必须有一个 README.md** | 当前为零入口，重构后必须有 |

### 🟡 可选/建议做的事

| # | 动作 | 理由 |
|---|------|------|
| 1 | **eddx 导出为 PNG/SVG 放入 assets/** | 方便非亿图用户查看，比赛时直接引用图片 |
| 2 | **mmd 文件同时保留在 assets/ 和 docs-planning/** | docs-planning/ 里放的是引用（如 `![流程图](../assets/core-workflow.png)`），assets/ 里放的是源文件 |
| 3 | **所有文档内部链接一次性校验** | 迁移后确保没有断链 |

## 四、内容合并策略（具体怎么合）

### 合并 1：业务目标

**来源**：`AAB业务目标/原本.md` + `译本.md`
**目标**：`docs-planning/01-business-goals.md`
**策略**：以译本为正文底本，在开头加一段「背景来源」引用原本的场景描述，然后删除原本.md。

### 合并 2：技术栈选型

**来源**：`AAG技术栈选型/` 下三层各一个 readme.md
**目标**：`docs-planning/06-tech-stack.md`
**策略**：

| 业务逻辑层 | FastAPI + Uvicorn | ... | ... |
| 数据访问层 | SQLite + aiosqlite | ... | ... |

## 表现层详细设计

[原 1.表现层/readme.md 的内容]

## 业务逻辑层详细设计

[原 2.业务逻辑层/readme.md 的内容]

## 数据访问层详细设计

### 合并 3：开发规范

**来源**：`开发模式组合/readme.md` + `TDD后端骨架+核心链路/readme.md`
**目标**：`docs-planning/08-development-guide.md`
**策略**：总览 + TDD 实践 + 核心链路设计，按章节拼接。

### 合并 4：功能清单

**来源**：`AAC功能清单/baseNeed.mmd` + `scorePie.mmd`
**目标**：`docs-planning/02-functional-requirements.md`
**策略**：mindmap 保留在 `assets/` 作为配图，正文用 Markdown 表格重新组织（功能点 + 赛题对应 + 评分权重列）。`scorePie.mmd` 的内容并入表格的权重列后删除。

### 迁移 5：架构设计

**来源**：`AAH架构设计/DevOpsAgent架构.eddx` + 空目录
**目标**：`assets/architecture-diagram.png` + `docs-planning/07-architecture.mmd`
**策略**：eddx 导出为 PNG 放入 assets，mindmap 用文字描述架构分层 + 引用图片。
