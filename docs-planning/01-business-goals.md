# 01 业务目标

> 项目背景、场景提炼与四大业务目标

---

## 背景

在企业级数据中心（如教育、医疗行业场景），运维人员常需处理复杂的 OS 问题（如：僵尸进程堆积、磁盘 I/O 异常、配置文件漂移）。传统运维依赖复杂的脚本和监控告警，门槛极高。

## 场景提炼

模拟一个真实的 Linux 生产节点，Agent 需应对管理员的自然语言诉求。

**示例**：当管理员说"帮我清理系统垃圾"时，Agent 必须通过环境感知发现是大日志文件占满了空间，但在执行删除前，必须触发"安全校验"，识别该文件是否为关键数据库日志，并评估当前权限是否合规，从而避免引发系统崩溃。

---

## 核心使命

构建一个面向国产化 Linux 生产环境的运维智能体，让管理员用自然语言就能完成 OS 运维操作，同时保证每一步操作都经过安全校验和权限管控，不会因误操作搞崩系统。

---

## 四大业务目标（按优先级）

### 目标 1 — 降低运维门槛

管理员不需要记命令、写脚本、查 man page，用中文说话就行。Agent 负责把自然语言翻译成正确的运维动作。

> 对应赛题：LLM 推理能力 + 对话交互

### 目标 2 — 自动环境感知

Agent 不是被动执行命令，而是主动理解当前系统状态再决策。用户说"清理垃圾"，Agent 自己去查磁盘、看进程、读日志，找到根因再动手。

> 对应赛题：OS 环境深度感知

### 目标 3 — 安全护栏

这是整个系统最关键的差异化能力。不是简单地说"我不能删 /"，而是要有语义级别的判断——这个文件是什么、删了会怎样、当前权限够不够。

> 对应赛题：安全意图校验器 + 最小权限执行

### 目标 4 — 国产化环境适配

跑在龙芯 loongarch64 + 麒麟 V11 上，不是 x86 + Ubuntu。所有依赖、工具链、执行路径都要兼容这套栈。

---

## 隐含非功能性需求

从场景简介里还隐含了几个没明说但必须做到的事：

### 可靠性要求

"避免引发系统崩溃"——意味着 Agent 不能猜，不能"看起来像对的就执行"，必须有确定性的校验链路。错了要么拒绝执行，要么给出明确警告让人类确认。

### 可追溯性

管理员说"帮我清理"，Agent 做了一系列操作——事后必须能回答"我做了什么、为什么这么做、依据是什么"。这就是五段式审计日志存在的业务理由。

### 领域适应性

教育和医疗场景的特点是——运维人员可能不是专职 Linux 管理员，而是兼岗的老师或信息科人员。所以"门槛极高"这个问题是真的痛点，不是虚设的场景。
